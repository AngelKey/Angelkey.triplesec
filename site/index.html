TripleSec

Governments wield great power. Consider:

  • the USA may compel websites & businesses to "cooperate"
  • the NSA may have already broken a popular cipher

If you're a programmer and wish to store user data that only the user can read -- excluding both you and governments -- you must encrypt in the browser or client app with care, and to avoid "provided" technologies such as a single encryption algorithm (possibly broken) or entropy provider (possibly intentionally weak).

TripleSec is a simple, open-source symmetric encryption library for the browser and Node.js. It aims for unbreakable encryption using Salsa 20, AES, and Twofish, so that a compromise of 1 (or even 2) of the ciphers will expose nothing. The 3 are combined in a technique (explained below) outlined by Bruce Schneier in Applied Cryptography.

In other words, TripleSec should work unless all 3 of have fallen. A dark, dark day that would be.

Installation

For the browser, download triplesec.js, then <script src="triplesec.js"></script>.

In node, npm install triplesec.

How to Use It

Encryption is performed by the encrypt function. It is non-blocking on the CPU and calls back with (err, buffer).

  • CoffeeScript
  • JavaScript
triplesec.encrypt ({

    data:          new triplesec.Buffer('Pssst. I believe I love you.'),
    key:           new triplesec.Buffer('top-secret-pw'),
    progress_hook: function (step, iterations, i) { /* ... */ }

}, function(err, buff) {
  
    if (! err) { 
        var ciphertext = buff.toString('hex');
    }

});

Algorithm Design

The Triplesec library encrypts data in four high-level steps:

  1. Key derivation. Given a user-provided password, and a random salt value, generate four separate secret keys, one for each cipher (see Step 3), and a final key for signing the ciphertext (see Step 4). This "key stretching" is done via PBKDF2 using the SHA-512 hash primitive. The output of this step is four seperate keys, used below.
  2. Initial value (IV) generation. A random number generator is queried to produce an initial for each of the three ciphers: a 192-bit IV for XSalsa20 (a slightly modified version of Salsa20 that takes a 192-bit nonce rather than the standard 64-bit nonce); a 128-bit IV for Twofish; and a 128-bit IV for AES.
  3. Cascading encryption. Each of the ciphers runs with the keys generated in Step 1, and the IVs generated in Step 2. The innermost cipher is XSalsa20, which is a stream cipher so doesn't need a block mode of operation. Given a key, and an IV, Salsa20 generates a random pad, which is then XOR'ed with the input message. This step of the algorithm then outputs the concatenation of the IV and the result of the XOR operation. This output is then the input for the next cipher, which is Twofish running in CTR mode. That is, given the Twofish key generated in Step 1 and the Twofish IV generated in Step 2 (call it R), an encrypted pad is output that is the size of the ciphertext output from the Salsa step. This pad will be of the concatenation of Twofish outputs when run on inputs R, R+1, R+2, etc, until enough pad is accumulated. Then, this pad is XOR'ed with the output from the Salsa20 step. The output of the Twofish step is the concatenation of the IV and the XOR result. The same process is repeated one last time, with AES also running in CTR mode.
  4. HMAC (or "sign") the ciphertext. Finally, Triplesec "signs" the ciphertext to ensure that it's not tampered by an adversary. The data to be signed is everything generated to date: a small header that encapsulates the version of the algorithm (now at 1); the salt used in key derivation; and the output of the AES stage of the chained encryption above. Triplesec uses HMAC with SHA-512 to prepare a mac. The final output is a concatenation of: the header; the salt; the signature; and the outermost ciphertext.

Though this is not the exact algorithm suggested by Schneier in Applied Cryptography (Section 15.8 in the Second Edition), it is close. Triplesec never uses the output of one block cipher as input into the next, which theoretically might allow a crack of one cipher to used to crack another. Rather, by merit of CTR mode, the three ciphers run on statistcally independent IVs, so a crack of one will not help the crack of the others. The Triplesec techniques takes one futher step not suggested by Schneier, which is the protect the inner IVs with the outer encryption algorithms, and only exposing the outermost IV in the clear. Though we can't prove this make the scheme more secure, it seems like a reasonable idea: why reveal cipher inputs if we don't have to. Finally, this algorithm has the added advantage in that the output ciphertext only increases by a constant additive amount (i.e., the lengths of the header, the salt, the HMAC and the three IVs). Schneier's technique inflates ciphertexts by a factor N, where N is the number of independent ciphers used.

What's triplesec.Buffer?

It is a wrapper around either Node.js's Buffer or a browser equivalent. When you generate encrypted data, you can use the output buffer however you like. In our above examples, we converted to and from hex strings.

How does TripleSec generate randomness/entropy? And can I change it?

In the browser, TripleSec uses window.crypto.getRandomValues and in Node it uses crypto.rng. For fear of a compromise -- let's say your browser author has inserted a weak version of one of these in exchange for some honeypot sex -- TripleSec seeds further with more-entropy (explained here in more detail), which does its own timing of a great number of math operations.
